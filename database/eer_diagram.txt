Relationship Cardinalities:
═══════════════════════════

• locations (1) ──→ (n) building_complexes
• locations (1) ──→ (n) customers (preferred_location)
• building_complexes (1) ──→ (n) apartments
• property_types (1) ──→ (n) apartments
• apartments (n) ──→ (n) features (through apartment_features)
• apartments (1) ──→ (n) sales
• apartments (1) ──→ (n) inquiries
• apartments (1) ──→ (n) viewings
• customers (1) ──→ (n) sales
• customers (1) ──→ (n) inquiries
• customers (1) ──→ (n) viewings
• agents (1) ──→ (n) sales
• agents (1) ──→ (n) inquiries
• agents (1) ──→ (n) viewings
• agents (1) ──→ (n) agents (self-referencing: manager_id)
• sales (1) ──→ (n) payment_schedule
• inquiries (1) ──→ (n) viewings

Key Design Features:
═══════════════════

✓ Normalized design (3NF) reduces data redundancy
✓ Comprehensive indexing for optimal query performance
✓ Flexible location hierarchy supports global operations
✓ Many-to-many apartment-features relationship
✓ Complete sales pipeline tracking (inquiry → viewing → sale)
✓ Payment schedule supports various payment methods
✓ Agent hierarchy supports management structures
✓ Status fields enable workflow management
✓ Audit fields (created_at, updated_at) for data tracking
✓ Soft delete capabilities through status fields

Business Rules Enforced:
═══════════════════════

• One apartment can only have one completed sale
• Unique apartment unit numbers within each building complex
• Unique customer email addresses and agent license numbers
• Commission amounts calculated from sale price and agent rate
• Payment schedules must sum to total sale price
• Viewings must be associated with prior inquiries
• Agents can manage other agents (hierarchical structure)
• Referential integrity maintained across all relationships
